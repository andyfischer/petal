# Migrating 100% to Bytecode — Removing the Graph Evaluator

**Status:** Phases 0–2 complete. Phase 3 next.

## Goal

Petal has two execution engines that live side by side (`rust/src/backend/`):

- **`graph`** — the original term-graph step evaluator. Reference engine.
- **`bytecode`** — a linear register VM that runs a lowering of the term graph.
  Already the `#[default]` backend, at full behavioral parity.

This project removes the graph *Evaluator* entirely, leaving bytecode as the
single execution engine.

## The distinction that shapes the work

"The old system" conflates two separate layers. Only one is removed:

| Layer | What it is | Fate |
|---|---|---|
| **Term-graph IR** (`program.rs`, Terms/Blocks) | Petal's identity — the dataflow substrate. Bytecode is a *lowering* of it. Slicing, IR-serialize, hot-reload state keys, autodiff-as-graph all reason about it. | **Stays. Untouched.** |
| **Graph step Evaluator** (`backend/graph/`) | The tree-walking interpreter that executes the term graph node-by-node. This is "the old path." | **Remove.** |

## What actually depends on the graph Evaluator

Most features assumed to need graph do not — they work under bytecode today:

- **Autodiff** — lives in shared `backend/ops.rs` (`a.derivative()`), used by
  both engines. ✅
- **Hot reload / state transfer** — `transfer_state.rs` already tests *both*
  backends. ✅
- **Slicing, IR-as-target, `show-ir`** — static analyses over `Program`,
  engine-independent. ✅

Only **two** things pin the graph Evaluator:

1. **Runtime trace buffer** → `explain` / `--trace` / `--record-trace` /
   `ExplainTerm` MCP tool. The graph Evaluator pushes `(term.id, inputs, result)`
   into `TraceBuffer` every step (`backend/graph/mod.rs:153`). The VM has no
   `trace` field, so it records nothing. `petal explain` enables trace and runs
   under the *default* (bytecode) backend — reasoning over a trace bytecode
   never populated.
2. **The correctness oracle.** The entire test strategy is differential against
   graph: `bytecode/tests.rs`, the 4-oracle `fuzz.rs`, `test-examples.ts`,
   `transfer_state.rs`. Deleting graph deletes the reference that proves
   bytecode correct.

## Decisions

- **Oracle: full parity first.** Build the oracle replacement (BC-noopt + a
  frozen golden corpus) *before* deleting graph, so the project is never in a
  state where correctness is unverifiable.
- **Trace feature: best-effort / can degrade.** `explain` / `ExplainTerm` stay
  *functional*, not byte-identical. In-place mutation and register reuse mean
  some intermediate values won't map cleanly to a TermId — that fidelity loss is
  acceptable. This removes the hardest part of the migration.

## Phases

Each phase is independently committable.

### Phase 0 — Baseline & golden corpus  *(done)*
- ✅ Baseline green: 257 Rust tests (`cargo test`), 24/24 `test-examples.ts`
  differential.
- ✅ Froze graph's `{status, stdout, stderr}` for every `examples/*.ptl` into a
  checked-in golden corpus: `test/example-golden/*.json`, generated by
  `ts/bin/gen-example-golden.ts` (run only while graph exists; regenerate
  deliberately — a golden update asserts the reference behavior itself changed).
- ✅ Verified bytecode already matches all 24 captures, so the corpus is a valid
  oracle.
- Note: the existing `test/<case>/expects` harness (`rust/tests/script_cases.rs`)
  already runs under the default (bytecode) backend — it is engine-independent
  and survives graph removal unchanged. No snippet-matrix freeze needed there;
  Phase 1 repoints the `bytecode/tests.rs` and `fuzz.rs` in-Rust oracles to
  BC-noopt.

### Phase 1 — Repoint the oracle (before any deletion)  *(done)*
- ✅ `fuzz.rs` — dropped the graph line from `assert_exact_parity`; the 3-way
  bytecode comparison (BC-noopt vs route-A vs BC-all) is now the oracle.
- ✅ `bytecode/tests.rs` — `run`/`run_stateful` take `OptFlags`;
  `assert_parity`/`assert_stateful_parity` diff `OptFlags::none` vs
  `OptFlags::all`. Collapsed the duplicate `run_opt` helper into `run`. Two
  direct graph uses repointed to BC opt levels.
- ✅ `transfer_state.rs` — expectations were already literal constants; swapped
  the `[Graph, Bytecode]` loop for `[OptFlags::none, OptFlags::all]` under
  bytecode.
- ✅ `test-examples.ts` — runs each example at `--no-opt` and default opts,
  requires them byte-identical, and checks both against the frozen
  `test/example-golden` corpus. Negative-tested (a corrupted golden fails).
- ✅ Verified: no `Backend::Graph` / `--backend=graph` remains in the four
  harnesses. Full suite green — 257 Rust tests, 3000-seed fuzz soak, 483 TS
  tests, 24/24 golden sweep. **Graph is no longer load-bearing for correctness.**
- Note: `ts/test/modules.test.ts` and `ts/bin/bench-backends.ts` still pass a
  `--backend` arg (benchmark + module-resolution harnesses, not correctness
  oracles) — handled in Phase 4 alongside the CLI flag removal.

### Phase 2 — VM trace (best-effort)  *(done)*
- ✅ Added `trace: &mut TraceBuffer` to `Vm`, wired from `Env::step_bytecode`.
- ✅ `Inst::dst()` / `Inst::input_regs()` helpers (`isa.rs`). Straight-line
  value-producing instructions push `(origin, inputs, result)` at retire in
  `Vm::step`, gated on `trace.enabled` (bool test first, so the disabled hot
  path never calls `dst()`/`input_regs()`). Calls are skipped there (dst not yet
  written) and traced from `Vm::deliver_value` against the call-site term.
- ✅ Fixed the regression: `petal explain` under the default (bytecode) backend
  showed `<not executed>` for every term; it now shows values.
- **Fidelity result** (better than the "can-degrade" bar): with opts off,
  bytecode `explain` is **byte-identical to graph**. The only divergence is with
  in-place mutation on — the trace stores heap ids, so historical entries for a
  mutated container display its *final* contents (e.g. a loop accumulator's
  early snapshots read as the finished list). This is inherent to zero-copy
  mutation and is the accepted best-effort gap.
- Tests: `bytecode_trace_records_term_values`,
  `bytecode_trace_records_call_results` in `bytecode/tests.rs`.

### Phase 3 — Delete the graph Evaluator
- Remove `backend/graph/`, `Backend::Graph`, `step_graph`,
  `--backend`/`PETAL_BACKEND`, `set_backend`; collapse `step_n`.
- Move shared items (`RuntimeClosure`, `StepResult`) out of `graph` into
  `backend/mod.rs`.
- Resolve `OptFlags` — keep it; BC-noopt is now the oracle.

### Phase 4 — Consumer & doc sweep
- `cli.rs` flag removal; de-parameterize the TS harnesses (`test-examples.ts`,
  `bench-backends.ts`, `modules.test.ts`).
- Update `backend/mod.rs`, `scripts.md`, `goals.md`, `Architecture.md`,
  `ir-as-target.md`.
- Confirm no `.ptl` scripts in `apps/`, `~/garden`, `~/.garden` are affected
  (engine-internal change — they shouldn't be).
