# Petal — Status & Goals — March 2026

## Current State

### Codebase

- **19 Rust source files**, ~6,660 lines
- **22 vitest test files**, 253 tests, all passing
- **16 example programs**, all passing
- Pipeline: Lexer → Parser → AST → Compiler → IR (term graph) → Step Evaluator
- Interactive playground (React + Vite + Express API)
- MCP tool server (4 tools: TestSnippet, ShowIR, ShowAST, ShowTokens)

### Architecture Strengths

- Clean pipeline with well-separated concerns
- Dataflow IR: term-graph with explicit `inputs` edges, organized into `Block`s
- `Value` is `Copy` via heap IDs — no `Rc<RefCell<...>>`
- Good Rust idioms: `SmallVec`, newtype IDs, no `unsafe`, `ControlFlow` enum
- Closures/captures handled correctly (with hard-won bug fixes)
- Mark-and-sweep GC with free-list slot reuse (triggers every 1024 allocations)
- Runtime string interning (deduplicates via intern table, cleaned on GC sweep)
- Source positions in all runtime errors (`[line N, column N]`)
- `is_callable` validation at parse time with clear error messages
- Name-based state keys for robust hot reload across code reordering

### Language Features

- **35 built-in functions**: print, range, len, push, str, abs, sqrt, floor, ceil, float, int, random, type, append, pop, keys, values, contains, min, max, round, dual, value_of, deriv_of, sort, reverse, join, split, enumerate, zip, slice, flat, map, filter, reduce
- **Native function FFI** — Lua-inspired `NativeFnTable` + `PetalCxt` handle system
- **Pipe operator** (`|>`) — `x |> f` → `f(x)`, `x |> f(y)` → `f(x, y)`
- **Method syntax** (`obj.method(args)`) — receiver becomes first arg
- **Compound assignment** — `+=`, `-=`, `*=`, `/=`, `%=`
- **String interpolation** — `"hello {name}"`
- **JSX-like element syntax** — `<Tag attr={expr}>children</Tag>`
- **`continue` statement** for loops
- **Forward-mode AD** — `Value::Dual` with chain rule through arithmetic and math builtins

## Goal Status

### Goal 1: Dataflow-First Language — ~40%

**Done:**
- IR is a real dataflow graph (terms with explicit input edges)
- Pipe operator provides surface-level dataflow syntax
- Method syntax for chaining
- Provenance querying API (`trace_provenance`, `trace_dependents`, `slice`)
- Forward-mode AD (dual numbers with chain rule)

**Not done:**
- No back-propagation through the computation graph
- No runtime querying or manipulation of the dataflow graph
- No "human-in-the-loop refinement" workflow

### Goal 2: First-Class State Management — ~70%

**Done:**
- `state` keyword with unique `StateKey`s
- State persists across invocations via `reset_stack()`
- `StateInit` only initializes if key isn't already set
- State read/write are proper IR terms in the dataflow graph
- Name-based state keys survive code reordering across hot reload

**Not done:**
- State inside loops doesn't give per-iteration state parameterized by loop index
- No "temporal edges" explicitly represented in the dataflow graph

### Goal 3: Projectional Views — ~20%

**Done:**
- Forward slice (`trace_dependents`) and backward slice (`slice`)
- DOT graph export with color-coded nodes

**Not done:**
- No projection API for scenario-based views
- No bidirectional editing, no cross-language projection

### Goal 4: Live Editing — ~40%

**Done:**
- Step-based evaluator with step-based loops
- `reset_stack()` preserves state while restarting execution
- `Env` separates programs from stacks
- Hot-reload prototype: recompile source, preserve matching state, drop removed, initialize new

**Not done:**
- No incremental recompilation
- No interactive "modify and continue" workflow

## Next Priorities

### High Impact
1. **Back-propagation / reverse-mode AD** — extend dual number system to support backward pass through the computation graph
2. **Incremental recompilation** — recompile only changed functions, keeping existing IR and state intact
3. **Per-iteration loop state** — state parameterized by loop index for stateful list rendering

### Medium Impact
4. **Runtime dataflow inspection** — API to query/traverse the live term graph from Petal code
5. **Projection API** — scenario-based views that show a subset of the dataflow graph
6. **Interactive modify-and-continue** — pause execution, edit code, resume with updated program
