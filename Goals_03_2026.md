### Step-Based Loops (was: "Loops break the step-based contract")

`ForLoop` and `WhileLoop` now use frame push/pop per iteration — they yield control back to the step evaluator on each iteration rather than running synchronously. This was the key architectural fix for Goal 4 (live editing).

### New Language Features

- **Pipe operator** (`|>`) — parse-time desugaring: `x |> f` → `f(x)`, `x |> f(y)` → `f(x, y)`
- **Method syntax** (`obj.method(args)`) — `TermOp::MethodCall`, receiver becomes first arg
- **Native function FFI** — Lua-inspired `NativeFnTable` + `PetalState` handle system
- **Compound assignment** — `+=`, `-=`, `*=`, `/=`, `%=`
- **String interpolation** — `"hello {name}"`
- **JSX-like element syntax** — `<Tag attr={expr}>children</Tag>`

- **24 built-in functions** (up from ~4 originally): print, range, len, push, str, abs, sqrt, floor, ceil, float, int, random, type, append, pop, keys, values, contains, min, max, round, map, filter, reduce

## Known Weaknesses (still open)

### No GC
`heap.rs` `collect()` is still a no-op stub. The heap only grows. Every string concatenation, list push, and range() call leaks. Long-running programs will eventually OOM.

### `apply_pattern_bindings` is O(N)
Still scans the entire `program.terms` vector to find terms in a given block. Should use a block→terms index.

### ~~No source positions in errors~~ ✅ DONE
Source positions now included in all runtime errors. Lexer tracks line/column, AST nodes carry `SourceSpan`, compiler populates `SourceMap`, evaluator annotates errors with `[line N, column N]`.

### No Rust unit tests
All testing is via the external vitest suite. No `#[test]` functions in any `.rs` file.

### `is_callable` is a stub
`parse.rs` — always returns `true`. `42(args)` parses as a call (fails at runtime, not parse time).

### No string interning at runtime
`heap.alloc_string()` creates a fresh allocation every time. The `ConstantTable` deduplicates compile-time constants, but runtime allocations are never interned.

## Goal Status

### Goal 1: Dataflow-First Language — ~20%

**Done:**
- IR is a real dataflow graph (terms with explicit input edges)
- Pipe operator provides surface-level dataflow syntax
- Method syntax for chaining

**Not done:**
- No provenance querying API
- No automatic differentiation / gradient types / chain rule
- No back-propagation through the computation graph
- No runtime querying or manipulation of the dataflow graph
- No "human-in-the-loop refinement" workflow

### Goal 2: First-Class State Management — ~70%

**Done:**
- `state` keyword with unique `StateKey`s
- State persists across invocations via `reset_stack()`
- `StateInit` only initializes if key isn't already set
- State read/write are proper IR terms in the dataflow graph

**Not done:**
- State inside loops doesn't give per-iteration state parameterized by loop index
- No "temporal edges" explicitly represented in the dataflow graph
- No state migration or structural comparison for live editing

### Goal 3: Projectional Views — ~5%

**Done:**
- SourceMap types exist (unpopulated)
- Dataflow graph structure could theoretically be sliced

**Not done:**
- No slicing algorithm, no projection API, no scenario-based views
- No bidirectional editing, no cross-language projection
- Essentially unstarted beyond having the right IR shape

### Goal 4: Live Editing — ~30%

**Done:**
- Step-based evaluator (correct architecture)
- Loops are now step-based (fixed since Feb 2024)
- `reset_stack()` preserves state while restarting execution
- `StateKey` provides correspondence between source locations and runtime state
- `Env` separates programs from stacks

**Not done:**
- No actual hot-reload mechanism (recompile + diff + reconcile)
- No incremental recompilation
- No state migration when code structure changes
- No interactive "modify and continue" workflow

## What Was Built Instead

Most development effort since Feb 2024 went into **language ergonomics and developer tooling** rather than the four research goals:

- Substantial new syntax (pipe, methods, compound assign, string interp, JSX, continue)
- Full CLI debugging toolchain (show-ir/ast/tokens with JSON output)
- Comprehensive test suite (144 tests)
- Web playground for interactive exploration
- MCP server for AI-assisted development
- Native FFI system
- 24 built-in functions (6x the original count)

This was arguably the right investment — the language needed to be usable and testable before the harder research features (AD, provenance, live editing) could be built on top of it. The foundation is now significantly more solid.

## Recommended Next Priorities

### High Impact, Moderate Effort
1. **Migrate builtins to native FFI** — The old `builtins.rs` system uses hardcoded `BuiltinId` constants and a 300-line `match` in `call_builtin()`. The new `native_fn.rs` FFI system (`NativeFnTable` + `PetalState` handle) is cleaner, extensible, and plugin-ready — but zero builtins have been migrated to it yet. Rewrite all 24 builtins as native functions, then remove the old `BuiltinTable`/`call_builtin` dispatch. This unifies the two calling conventions and proves the FFI is production-ready.
2. ~~**Source positions in errors**~~ ✅ DONE — Lexer tracks line/column, AST carries `SourceSpan`, compiler calls `source_map.add()`, evaluator annotates errors.
3. **Garbage collection** — Implement mark-and-sweep using the existing `_gc_mark` fields. The infrastructure is already scaffolded.
4. **Block→terms index** — Replace the O(N) scan in `apply_pattern_bindings` with a `HashMap<BlockId, Vec<TermId>>` built once at compile time.

### Medium Impact, Lower Effort
5. **Runtime string interning** — Add a hash-set dedup layer to `alloc_string()`.
6. **`is_callable` validation** — Restrict callable expressions at parse time.

### Research Goals (High Effort)
7. **Provenance queries** — Add an API to trace backward through the term graph from any output value to its inputs. The IR already supports this.
8. **Hot reload prototype** — Recompile source, diff the new program against the old one, reconcile state keys.
9. **Forward-mode AD** — Add gradient Value types and chain-rule propagation through arithmetic terms.
