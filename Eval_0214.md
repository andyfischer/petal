# Petal Implementation Evaluation — 2024-02-14

## Codebase Quality Assessment

### Strengths

**Architecture is clean and well-separated.** The pipeline (Lexer → Parser → AST → Compiler → IR → Step Evaluator) across 14 files / ~4,850 lines is well-organized. Each module has a clear responsibility with minimal leakage between concerns.

**The IR design is genuinely thoughtful.** The term-graph representation — where every computation is a `Term` with explicit `inputs: SmallVec<[TermId; 4]>` dataflow edges, organized into `Block`s — is not a bytecode VM or tree-walker. It's a real dataflow graph that preserves the structure the goals document demands. This was a harder architectural choice that pays dividends later.

**Value representation is smart.** Making `Value` `Copy` by using heap IDs (StringId, ListId, MapId) instead of `Rc<RefCell<...>>` avoids an entire class of borrow-checker struggles. The register-file approach (Vec<Value> per frame) is cache-friendly and simple.

**Good Rust idioms.** `SmallVec` for hot paths, newtype IDs for type safety (`TermId`, `BlockId`, `RegisterIndex`), no `unsafe`, proper error propagation. The `ControlFlow` enum for evaluator signals is clean.

**Closures and captures are handled correctly** — and the bug history (FunctionId collision, phantom block placement, guard pattern vars) shows this was hard-won correctness.

### Weaknesses

**No GC.** `heap.rs:109` has `collect()` as a no-op stub. The heap only grows. Every string concatenation, every list `push`, every `range()` call leaks. The Game of Life example running for more generations would OOM eventually.

**Loops break the step-based contract.** `ForLoop` (eval.rs) and `WhileLoop` (eval.rs) run their bodies to completion *synchronously* inside `exec_term`, calling `Self::step()` recursively. This means a for-loop over 10,000 items is 10,000 nested steps that can't be interrupted. This directly undermines the step-based architecture that Goal 4 (live editing) depends on.

**`apply_pattern_bindings` is O(N) over all terms.** It scans the *entire* `program.terms` vector to find terms in a given block. For large programs this becomes a bottleneck on every match arm execution.

**No source positions in errors.** The `SourceMap` infrastructure exists but is never populated during compilation and never used in error reporting. All errors are just bare strings like `"Division by zero"` with no line numbers.

**No unit tests.** The only testing is the 16 `.ptl` example files run through `test-each.sh`. There are no `#[test]` functions, no edge-case tests for the lexer/parser/compiler, no regression tests for the three bugs already found and fixed.

**`is_callable` is a stub** (parse.rs) — always returns `true` regardless of expression type. The comment suggests intent that was never implemented.

**String allocation is not interned.** The `ConstantTable` deduplicates compile-time constants, but runtime `heap.alloc_string()` creates a fresh allocation every time — even for identical strings from repeated operations.

## How Well the Implementation Fulfills the Goals

### Goal 1: Dataflow-First Language — Foundation laid, ~15% of vision

**What's done:**
- The IR is a real dataflow graph. Every `Term` has explicit input edges to the terms it depends on. You *can* trace from any output term backward through its inputs to find the complete chain of expressions that produced it. This is the right data structure.
- The compiler produces this graph from source, and the evaluator executes it.

**What's missing:**
- No provenance querying API ("what inputs influenced this output?")
- No automatic differentiation — no gradient types, no chain rule implementation, no forward or reverse mode AD
- No back-propagation through the computation graph
- No mechanism for the "human-in-the-loop refinement" workflow described in the goals
- The dataflow graph is write-once at compile time — there's no infrastructure for dynamically querying or manipulating it at runtime

The IR *could* support these features, but none of the actual capabilities from Goal 1 (provenance queries, AD, back-propagation) exist yet. The implementation is a conventional language interpreter with an unconventional IR.

### Goal 2: First-Class State Management — Substantially implemented, ~70% of vision

**What's done:**
- The `state` keyword works correctly with unique `StateKey`s assigned at parse time
- State persists across invocations via `reset_stack()` — this is the core promise
- `StateInit` only initializes if the key isn't already set (preserving state across re-runs)
- State in loops gets per-iteration keys (though by parser assignment, not by iteration index)
- State read/write are proper IR terms (`StateInit`, `StateRead`, `StateWrite`) — they *are* part of the dataflow graph, not a side-channel
- The `state.ptl`, `state_machine.ptl`, and `fibonacci.ptl` examples demonstrate this working

**What's missing:**
- State inside loops doesn't truly give "per-iteration state" in the way the goals describe (animated_particles example). The StateKey is fixed at parse time, not parameterized by loop index. Running the loop twice would use the same state keys.
- "Temporal edges" in the dataflow graph (connecting output of one invocation to input of the next) aren't explicitly represented — the state HashMap is outside the graph
- No state migration or structural comparison for live editing scenarios

### Goal 3: Projectional Views — Infrastructure only, ~5% of vision

**What's done:**
- `SourceMap` exists with `SourceSpan` types (though it's never populated)
- The dataflow graph structure could, in principle, be sliced forward or backward

**What's missing:**
- No slicing algorithm (forward or backward)
- No projection API
- No scenario-based views
- No bidirectional editing
- No cross-language projection
- None of the tooling described in the goals (querying provenance, computing projections statically or dynamically)

This goal is essentially untouched beyond having the right IR shape.

### Goal 4: Live Editing — Architecture supports it, ~20% of vision

**What's done:**
- The step-based evaluator is the correct architecture — it enables pausing execution at any point
- `reset_stack()` preserves state while restarting execution, which is the core of "state reconciliation"
- `StateKey` provides the "correspondence between source locations and runtime state" that the goals describe
- The `Env` separates programs from stacks, allowing multiple programs and re-loading

**What's missing:**
- No actual hot-reload mechanism (recompile + diff + reconcile)
- Loops execute synchronously, which defeats step-by-step control during loop execution
- No incremental recompilation (would need to diff old and new term graphs)
- No state migration when code structure changes
- No interactive "modify and continue" workflow

## Summary

| Goal | Status | Score |
|------|--------|-------|
| 1. Dataflow-First | Correct IR shape, no querying/AD/provenance | ~15% |
| 2. First-Class State | Working `state` keyword, persistence across runs | ~70% |
| 3. Projectional Views | Essentially unstarted | ~5% |
| 4. Live Editing | Right architecture, no actual live-edit flow | ~20% |

**Overall: The implementation is a solid, well-crafted interpreter for a stateful language with a dataflow-oriented IR.** The code quality is good — it's compact, idiomatic Rust, and all 16 examples work. But it's in the "language that could eventually support the vision" stage rather than "language that delivers the vision." The IR design was the right bet to make first — it's much easier to add provenance queries and AD on top of a term graph than to retrofit them onto a bytecode VM. The most urgent gaps are practical ones (no GC, synchronous loops, no source positions in errors, no tests) and the big conceptual features (differentiation, projection, live editing) are future work that the foundation can support.
