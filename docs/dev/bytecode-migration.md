# Migrating 100% to Bytecode ‚Äî Removing the Graph Evaluator

**Status:** Phase 0 + Phase 1 complete. Phase 2 next.

## Goal

Petal has two execution engines that live side by side (`rust/src/backend/`):

- **`graph`** ‚Äî the original term-graph step evaluator. Reference engine.
- **`bytecode`** ‚Äî a linear register VM that runs a lowering of the term graph.
  Already the `#[default]` backend, at full behavioral parity.

This project removes the graph *Evaluator* entirely, leaving bytecode as the
single execution engine.

## The distinction that shapes the work

"The old system" conflates two separate layers. Only one is removed:

| Layer | What it is | Fate |
|---|---|---|
| **Term-graph IR** (`program.rs`, Terms/Blocks) | Petal's identity ‚Äî the dataflow substrate. Bytecode is a *lowering* of it. Slicing, IR-serialize, hot-reload state keys, autodiff-as-graph all reason about it. | **Stays. Untouched.** |
| **Graph step Evaluator** (`backend/graph/`) | The tree-walking interpreter that executes the term graph node-by-node. This is "the old path." | **Remove.** |

## What actually depends on the graph Evaluator

Most features assumed to need graph do not ‚Äî they work under bytecode today:

- **Autodiff** ‚Äî lives in shared `backend/ops.rs` (`a.derivative()`), used by
  both engines. ‚úÖ
- **Hot reload / state transfer** ‚Äî `transfer_state.rs` already tests *both*
  backends. ‚úÖ
- **Slicing, IR-as-target, `show-ir`** ‚Äî static analyses over `Program`,
  engine-independent. ‚úÖ

Only **two** things pin the graph Evaluator:

1. **Runtime trace buffer** ‚Üí `explain` / `--trace` / `--record-trace` /
   `ExplainTerm` MCP tool. The graph Evaluator pushes `(term.id, inputs, result)`
   into `TraceBuffer` every step (`backend/graph/mod.rs:153`). The VM has no
   `trace` field, so it records nothing. `petal explain` enables trace and runs
   under the *default* (bytecode) backend ‚Äî reasoning over a trace bytecode
   never populated.
2. **The correctness oracle.** The entire test strategy is differential against
   graph: `bytecode/tests.rs`, the 4-oracle `fuzz.rs`, `test-examples.ts`,
   `transfer_state.rs`. Deleting graph deletes the reference that proves
   bytecode correct.

## Decisions

- **Oracle: full parity first.** Build the oracle replacement (BC-noopt + a
  frozen golden corpus) *before* deleting graph, so the project is never in a
  state where correctness is unverifiable.
- **Trace feature: best-effort / can degrade.** `explain` / `ExplainTerm` stay
  *functional*, not byte-identical. In-place mutation and register reuse mean
  some intermediate values won't map cleanly to a TermId ‚Äî that fidelity loss is
  acceptable. This removes the hardest part of the migration.

## Phases

Each phase is independently committable.

### Phase 0 ‚Äî Baseline & golden corpus  *(done)*
- ‚úÖ Baseline green: 257 Rust tests (`cargo test`), 24/24 `test-examples.ts`
  differential.
- ‚úÖ Froze graph's `{status, stdout, stderr}` for every `examples/*.ptl` into a
  checked-in golden corpus: `test/example-golden/*.json`, generated by
  `ts/bin/gen-example-golden.ts` (run only while graph exists; regenerate
  deliberately ‚Äî a golden update asserts the reference behavior itself changed).
- ‚úÖ Verified bytecode already matches all 24 captures, so the corpus is a valid
  oracle.
- Note: the existing `test/<case>/expects` harness (`rust/tests/script_cases.rs`)
  already runs under the default (bytecode) backend ‚Äî it is engine-independent
  and survives graph removal unchanged. No snippet-matrix freeze needed there;
  Phase 1 repoints the `bytecode/tests.rs` and `fuzz.rs` in-Rust oracles to
  BC-noopt.

### Phase 1 ‚Äî Repoint the oracle (before any deletion)  *(done)*
- ‚úÖ `fuzz.rs` ‚Äî dropped the graph line from `assert_exact_parity`; the 3-way
  bytecode comparison (BC-noopt vs route-A vs BC-all) is now the oracle.
- ‚úÖ `bytecode/tests.rs` ‚Äî `run`/`run_stateful` take `OptFlags`;
  `assert_parity`/`assert_stateful_parity` diff `OptFlags::none` vs
  `OptFlags::all`. Collapsed the duplicate `run_opt` helper into `run`. Two
  direct graph uses repointed to BC opt levels.
- ‚úÖ `transfer_state.rs` ‚Äî expectations were already literal constants; swapped
  the `[Graph, Bytecode]` loop for `[OptFlags::none, OptFlags::all]` under
  bytecode.
- ‚úÖ `test-examples.ts` ‚Äî runs each example at `--no-opt` and default opts,
  requires them byte-identical, and checks both against the frozen
  `test/example-golden` corpus. Negative-tested (a corrupted golden fails).
- ‚úÖ Verified: no `Backend::Graph` / `--backend=graph` remains in the four
  harnesses. Full suite green ‚Äî 257 Rust tests, 3000-seed fuzz soak, 483 TS
  tests, 24/24 golden sweep. **Graph is no longer load-bearing for correctness.**
- Note: `ts/test/modules.test.ts` and `ts/bin/bench-backends.ts` still pass a
  `--backend` arg (benchmark + module-resolution harnesses, not correctness
  oracles) ‚Äî handled in Phase 4 alongside the CLI flag removal.

### Phase 2 ‚Äî VM trace (best-effort)
- Add `trace: &mut TraceBuffer` to `Vm`; emit `(origin, inputs, result)` at
  instruction retire, gated on `trace.enabled`, using the existing
  `func.origins: Vec<Option<TermId>>` table (already fetched every step for
  error annotation, `vm.rs:184`).
- Accept fidelity gaps. Keep `explain` / `ExplainTerm` functional.

### Phase 3 ‚Äî Delete the graph Evaluator
- Remove `backend/graph/`, `Backend::Graph`, `step_graph`,
  `--backend`/`PETAL_BACKEND`, `set_backend`; collapse `step_n`.
- Move shared items (`RuntimeClosure`, `StepResult`) out of `graph` into
  `backend/mod.rs`.
- Resolve `OptFlags` ‚Äî keep it; BC-noopt is now the oracle.

### Phase 4 ‚Äî Consumer & doc sweep
- `cli.rs` flag removal; de-parameterize the TS harnesses (`test-examples.ts`,
  `bench-backends.ts`, `modules.test.ts`).
- Update `backend/mod.rs`, `scripts.md`, `goals.md`, `Architecture.md`,
  `ir-as-target.md`.
- Confirm no `.ptl` scripts in `apps/`, `~/garden`, `~/.garden` are affected
  (engine-internal change ‚Äî they shouldn't be).
