### Step-Based Loops (was: "Loops break the step-based contract")

`ForLoop` and `WhileLoop` now use frame push/pop per iteration — they yield control back to the step evaluator on each iteration rather than running synchronously. This was the key architectural fix for Goal 4 (live editing).

## Known Weaknesses (still open)

### ~~No GC~~ ✅ DONE
Mark-and-sweep GC implemented. Heap objects (strings, lists, maps, elements) are traced from roots (stack registers, state, closures, loop state) and unmarked objects are reclaimed. Free lists enable slot reuse. Collection triggers automatically after every 1024 allocations.

### ~~`apply_pattern_bindings` is O(N)~~ ✅ DONE
Replaced with `HashMap<BlockId, Vec<TermId>>` index built once at compile time. Pattern binding now does O(B) lookup where B is the block's term count, instead of scanning all N program terms.

### ~~No source positions in errors~~ ✅ DONE
Source positions now included in all runtime errors. Lexer tracks line/column, AST nodes carry `SourceSpan`, compiler populates `SourceMap`, evaluator annotates errors with `[line N, column N]`.

### ~~No Rust unit tests~~ ✅ DONE
40 `#[test]` functions across lexer.rs (15 tests), value.rs (10 tests), program.rs (12 tests), and env.rs (3 tests) covering tokenization, truthiness, type names, float formatting, term lookup, provenance tracing, forward slicing, and hot reload.

### ~~`is_callable` is a stub~~ ✅ DONE
`check_callable()` now validates at parse time. Literals, binary/unary ops, lists, records, string interpolations, and elements are rejected with clear errors. Identifiers, field/index access, calls, lambdas, blocks, if/match are allowed.

### ~~No string interning at runtime~~ ✅ DONE
`alloc_string()` now checks a `HashMap<String, StringId>` intern table before allocating. Duplicate strings reuse existing heap slots. The intern table is cleaned during GC sweep. String equality can fast-path on `StringId` comparison.

## Goal Status

### Goal 1: Dataflow-First Language — ~40%

**Done:**
- IR is a real dataflow graph (terms with explicit input edges)
- Pipe operator provides surface-level dataflow syntax
- Method syntax for chaining
- Provenance querying API (`trace_provenance`) — traces backward through term inputs to find all ancestors and edges. CLI: `petal show-provenance --term <name> [--json] <file>`. Query by name or term id.

**Not done:**
- ~~No automatic differentiation / gradient types / chain rule~~ ✅ Forward-mode AD done (dual numbers with chain rule through arithmetic)
- No back-propagation through the computation graph
- No runtime querying or manipulation of the dataflow graph
- No "human-in-the-loop refinement" workflow

### Goal 2: First-Class State Management — ~70%

**Done:**
- `state` keyword with unique `StateKey`s
- State persists across invocations via `reset_stack()`
- `StateInit` only initializes if key isn't already set
- State read/write are proper IR terms in the dataflow graph

**Not done:**
- State inside loops doesn't give per-iteration state parameterized by loop index
- No "temporal edges" explicitly represented in the dataflow graph
- ~~No state migration or structural comparison for live editing~~ ✅ State keys are now name-based (hash of variable name), so reordering state declarations preserves state across hot reload

### Goal 3: Projectional Views — ~20%

**Done:**
- SourceMap types exist (unpopulated)
- Dataflow graph structure could theoretically be sliced
- Forward slice (`trace_dependents`) — traces all terms transitively affected by a given term. CLI: `petal show-dependents --term <name> [--json] <file>`
- Backward slice (`slice`) — computes minimal subgraph needed to compute target terms, returned in topological order. CLI: `petal show-slice --term <name> [--term <name2>] [--json] <file>`
- DOT graph export (`show-graph`) — outputs dataflow graph in Graphviz DOT format with color-coded nodes (constants=blue, state=yellow, calls=green, branches=salmon, loops=plum, closures=coral)

**Not done:**
- No projection API for scenario-based views
- No bidirectional editing, no cross-language projection

### Goal 4: Live Editing — ~40%

**Done:**
- Step-based evaluator (correct architecture)
- Loops are now step-based (fixed since Feb 2024)
- `reset_stack()` preserves state while restarting execution
- `StateKey` provides correspondence between source locations and runtime state
- `Env` separates programs from stacks
- Hot-reload prototype: `Env::hot_reload()` recompiles source, preserves state with matching StateKeys, drops state for removed keys, initializes new state

**Not done:**
- No incremental recompilation
- ~~No state migration when code structure changes (reordered state declarations get new keys)~~ ✅ Name-based state keys now survive reordering
- No interactive "modify and continue" workflow

## What Was Built Instead

Most development effort since Feb 2024 went into **language ergonomics and developer tooling** rather than the four research goals:

- Substantial new syntax (pipe, methods, compound assign, string interp, JSX, continue)
- Full CLI debugging toolchain (show-ir/ast/tokens with JSON output)
- Comprehensive test suite (144 tests)
- Web playground for interactive exploration
- MCP server for AI-assisted development
- Native FFI system
- 24 built-in functions (6x the original count)

This was arguably the right investment — the language needed to be usable and testable before the harder research features (AD, provenance, live editing) could be built on top of it. The foundation is now significantly more solid.
